Resumen Ejecutivo

Visión general del proyecto y estado actual

¿Qué es MAGNA CRM?

MAGNA CRM es una plataforma premium de gestión de relaciones con clientes (CRM) diseñada específicamente para Buyers Agents inmobiliarios. Centraliza toda la operativa comercial: cartera de clientes, propiedades, pipeline de ventas, colaboradores, calendario y analíticas, todo desde un único sistema web moderno y accesible desde cualquier dispositivo.

✅ Frontend 100% Terminado

El sistema cuenta con una interfaz completamente construida: 10 módulos funcionales, diseño responsivo para móvil/tablet/escritorio, modo oscuro, animaciones y más de 51 componentes UI premium. El trabajo de desarrollo visual está completado.

⚡ Siguiente paso: Conectar Backend y Base de Datos

Actualmente los datos son de demostración (mock data). El plan de implementación que se detalla en este documento convierte el CRM en un sistema real y persistente conectado a Supabase (PostgreSQL en la nube), con autenticación segura y despliegue en producción.

5.2 — SQL Completo de Creación (Supabase)

El siguiente SQL se ejecuta directamente en el SQL Editor de Supabase para crear todas las tablas del sistema:

-- ===================================================== -- MAGNA CRM — Esquema completo Supabase (PostgreSQL) -- Ejecutar en: supabase.com → SQL Editor -- ===================================================== -- 1. PERFILES DE USUARIO (extiende auth.users) CREATE TABLE profiles ( id UUID REFERENCES auth.users(id) PRIMARY KEY, full_name TEXT, avatar_url TEXT, role TEXT DEFAULT 'agent', -- admin | agent | manager workspace_name TEXT, created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW() ); -- 2. CLIENTES CREATE TABLE clients ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, name TEXT NOT NULL, email TEXT, phone TEXT, budget DECIMAL(12,2), zone TEXT, status TEXT DEFAULT 'activo', -- activo | templado | dormido | cerrado origin TEXT, -- referido | web | eventos | redes | otro client_type TEXT, -- comprador | inversor | propietario family_size TEXT, score INTEGER DEFAULT 50, notes TEXT, avatar_url TEXT, assigned_agent_id UUID REFERENCES profiles(id), created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW() ); -- 3. INTERACCIONES CON CLIENTES CREATE TABLE client_interactions ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, client_id UUID REFERENCES clients(id) ON DELETE CASCADE, type TEXT, -- llamada | mensaje | reunion | email description TEXT, date TIMESTAMPTZ DEFAULT NOW(), created_by UUID REFERENCES profiles(id) ); -- 4. COLABORADORES CREATE TABLE collaborators ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, agency_name TEXT NOT NULL, contact_name TEXT, email TEXT, phone TEXT, city TEXT, specialty TEXT, rating DECIMAL(3,1) DEFAULT 0, notes TEXT, created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW() ); -- 5. HISTORIAL DE COLABORADORES CREATE TABLE collaborator_history ( id UUID DEFAULT gen_random_uuid() PRIMARY KEY, collaborator_id UUID REFERENCES collaborators(id) ON DELETE CASCADE, event TEXT, date TIMESTAMPTZ DEFAULT NOW() );

5.3 — Seguridad: Row Level Security (RLS)

Supabase usa RLS (Row Level Security) para que cada agente acceda únicamente a sus propios datos:

-- Activar RLS en todas las tablas ALTER TABLE profiles ENABLE ROW LEVEL SECURITY; ALTER TABLE clients ENABLE ROW LEVEL SECURITY; ALTER TABLE client_interactions ENABLE ROW LEVEL SECURITY; ALTER TABLE collaborators ENABLE ROW LEVEL SECURITY; ALTER TABLE properties ENABLE ROW LEVEL SECURITY; ALTER TABLE pipeline_deals ENABLE ROW LEVEL SECURITY; ALTER TABLE tasks ENABLE ROW LEVEL SECURITY; ALTER TABLE events ENABLE ROW LEVEL SECURITY; -- Ejemplo: agente solo ve clientes asignados a él CREATE POLICY "agents_own_clients" ON clients FOR ALL USING (assigned_agent_id = auth.uid());

Plan de Implementación

5 fases ordenadas para llevar el CRM a producción real

1
Configuración Supabase y Base de Datos
2–3 días
  • Crear cuenta y proyecto en supabase.com
  • Ejecutar el SQL completo de las 12 tablas en el Editor SQL
  • Configurar Row Level Security (RLS) para seguridad multi-agente
  • Habilitar autenticación por email + contraseña
  • Crear bucket de almacenamiento (Storage) para avatares y fotos
  • Copiar variables de entorno: NEXT_PUBLIC_SUPABASE_URL y SUPABASE_ANON_KEY
  • Añadir variables al archivo .env del proyecto
2
API Routes — Backend CRUD
3–5 días
  • Instalar paquetes: @supabase/supabase-js y @supabase/ssr
  • Crear cliente Supabase centralizado en src/lib/supabase.ts
  • Crear endpoint /api/clients (GET, POST, PUT, DELETE)
  • Crear endpoint /api/properties (GET, POST, PUT, DELETE)
  • Crear endpoint /api/pipeline (GET, POST, PUT)
  • Crear endpoint /api/tasks (GET, POST, PUT, DELETE)
  • Crear endpoint /api/events (GET, POST, PUT, DELETE)
  • Crear endpoint /api/collaborators (GET, POST, PUT, DELETE)
3
Autenticación Real (Supabase Auth)
1–2 días
  • Reemplazar login simulado por supabase.auth.signInWithPassword()
  • Configurar middleware Next.js para proteger todas las rutas
  • Gestión de sesión persistente con cookies SSR (seguro)
  • Logout con supabase.auth.signOut()
  • Perfil de usuario cargado desde tabla profiles
  • Soporte para múltiples agentes y roles (admin, agente, manager)
4
Integración Frontend con Backend Real
3–5 días
  • Reemplazar datos mock (data.ts) por llamadas a las API routes con React Query
  • Formularios de alta y edición funcionales con Zod + React Hook Form
  • Notificaciones en tiempo real usando Supabase Realtime channels
  • Upload de avatares y fotos de propiedades a Supabase Storage
  • Paginación y búsqueda conectadas a la base de datos real
  • Estado de carga y manejo de errores en toda la UI
5
Despliegue a Producción
1–2 días
  • Deploy en Vercel (ya configurado con vercel.json)
  • Configurar variables de entorno en el dashboard de Vercel
  • Vincular dominio personalizado (.com) con SSL automático
  • Activar proyecto Supabase en modo producción
  • Habilitar backups automáticos diarios en Supabase
  • Pruebas end-to-end del sistema completo
  • Monitorización con Vercel Analytics y Supabase Logs
